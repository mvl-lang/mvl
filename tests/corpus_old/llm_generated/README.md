# LLM-Generated MVL Corpus

Corpus of MVL programs generated by LLMs (Claude, GPT, etc.) to study
real-world failure modes and measure self-healing capability.

**Related issue:** #685 | **Parent epic:** #669

---

## Structure

```
llm_generated/
├── README.md          — this file
├── schema.yaml        — YAML schema for attempt records
├── successful/        — programs that compiled on the first attempt
├── self_healed/       — programs fixed after receiving error feedback
├── failed/            — programs that could not be fixed
└── analysis/
    ├── common_mistakes.md    — recurring error patterns
    └── error_message_gaps.md — errors LLMs struggle to understand
```

## File naming

Each program is stored as `NNN_description.mvl`.  When the LLM needed
multiple attempts, all attempts are captured in `NNN_description.attempt.yaml`
(see `schema.yaml` for the field definitions).

Examples:
```
successful/001_safe_divide.mvl
self_healed/002_bounded_loop.mvl
self_healed/002_bounded_loop.attempt.yaml
failed/003_complex_refinement.mvl
failed/003_complex_refinement.attempt.yaml
```

## Collection process

1. Give an LLM a natural-language task (e.g. "Write a safe integer divider").
2. Run `mvl check` on the output.
3. If it fails, feed the error back to the LLM and retry.
4. Record each attempt in the `.attempt.yaml` file.
5. File in `successful/`, `self_healed/`, or `failed/` accordingly.

## CI integration

- Files in `successful/` and `self_healed/` must pass `mvl check` (no annotation needed).
- Files in `failed/` must be annotated with `// corpus:expect-fail` so that
  `make test-corpus` correctly expects them to fail.

## Metrics to track

| Metric | Description |
|--------|-------------|
| First-attempt success rate | % that pass `mvl check` immediately |
| Self-healing rate | % fixed after ≤ 3 error-feedback rounds |
| Average attempts to success | Mean over self_healed/ programs |
| Unrecoverable failure rate | % in failed/ after 5+ attempts |

See `analysis/` for accumulated findings.
