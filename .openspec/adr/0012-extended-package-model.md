# ADR-0012: Extended Package Model — Extern Inside, Verified API Outside

**Status:** Accepted
**Date:** 2026-04-15
**Context:** MVL needs a package ecosystem for third-party libraries. Every extended package wraps native Rust code (via `extern "rust"`) behind a fully-verified MVL API. This ADR answers the six design questions from issue #57 and formalises the package model that ADR-0007 and ADR-0009 already reference as `use pkg.*`.

---

## The Pattern

```
mvl_http/
├── mvl.toml                  # package manifest
├── src/
│   ├── server.mvl            # public API — full 11-requirement verification
│   ├── request.mvl           # Request with Tainted body
│   ├── response.mvl          # Response with Clean body, refined status
│   └── internal/
│       └── ffi.mvl           # extern "rust" { hyper } — hidden from users
└── bridge.rs                 # Rust implementations of extern fns
```

**Extern at the bottom, verified API at the top.** The library author takes responsibility for the trust boundary. User code receives full 11-requirement verification against the package's public API.

---

## Decision

### 1. Package Format — `mvl.toml` with `[dependencies]`

Packages use the same `mvl.toml` manifest already established in ADR-0009 for projects. No second file format. The `[dependencies]` table lists MVL packages; `[native]` (optional) tracks Rust crates required by bridge.rs.

```toml
[package]
name    = "http"
version = "1.2.0"
license = "MIT"
requires-mvl = ">=0.24.0"

# Why you can't generate it — required for any extern block
extern-rationale = "wraps hyper for async HTTP; no pure-MVL HTTP stack yet"

[dependencies]
tls = { git = "https://github.com/lab271/mvl_tls", tag = "v0.4.0" }

[native]
# Rust crates used in bridge.rs — appears in SBOM
hyper = "1.0"
tokio = { version = "1.0", features = ["full"] }
```

**Required fields:** `name`, `version`, `license`, `requires-mvl`.
**Required when any `extern` block exists:** `extern-rationale`.
**SBOM note:** `[native]` entries feed directly into the SBOM generated by `mvl build --sbom` (ADR-0003 / #56).

### 2. Visibility — `internal/` Convention + `pub` Keyword

Both mechanisms work together; they are complementary, not alternatives.

- **`pub` keyword** controls item-level visibility (functions, types, constants). Mirrors the module system (Spec 005 / ADR modules).
- **`internal/` directory** makes an entire subtree private to the package. Files under `src/internal/` are never re-exported through the package API surface, even if individual items are marked `pub` within them. This matches the Go convention and is easy to audit without reading every file.

The rule:

| Location | Default visibility | Can be `pub`? | Exported to users? |
|---|---|---|---|
| `src/*.mvl` | Public (package API) | Yes | Yes |
| `src/internal/*.mvl` | Package-private | Yes (within package) | **No** |

The compiler MUST reject `use pkg.http.internal.ffi` from user code. `internal/` is a hard boundary enforced at resolve time, not just a convention.

### 3. Registry — Git-Only (Phase 3), Central Registry (Phase 4)

**Phase 3:** Dependencies are declared with `{ git = "...", tag = "..." }`. No central registry. Simple, reproducible, avoids ecosystem lock-in during language evolution.

**Phase 4:** Central registry at `registry.mvl-lang.org` (crates.io model). Packages publish with `mvl publish`. Short names (`tls = "0.4"`) resolve through the registry. The git form remains valid forever.

Rationale: rushing a registry before the language stabilises leads to namespace squatting and API churn. Git-first mirrors the early Go modules model, which worked well.

### 4. Versioning — Semver Enforced, Type-Change Detection

- `version` in `mvl.toml` MUST follow semver (`MAJOR.MINOR.PATCH`).
- `mvl build` rejects packages whose `requires-mvl` range excludes the current compiler.
- **Breaking change detection (Phase 4):** Because MVL types are structural, the checker can compare a package's exported type signatures against the previous version and flag additions/removals in public API. A removed exported function or a changed type in a public signature is a detectable breaking change. `mvl publish` will warn when public API changes without a major bump.
- `mvl.lock` pins exact versions for reproducible builds (same model as Cargo.lock).

### 5. Build Integration — `mvl build` Resolves and Fetches

No separate `mvl install`. `mvl build` does everything:

```
mvl build
  1. Read mvl.toml + mvl.lock
  2. Fetch missing packages to $XDG_CACHE_HOME/mvl/pkg/
  3. Resolve use pkg.* imports to fetched sources
  4. Transpile: project source + package sources → single Rust crate
  5. Inject bridge.rs files (project + packages)
  6. cargo build
```

Local overrides go in `.mvl/pkg/<name>/` and take precedence over the cache. This enables patching a dependency without forking.

`mvl.lock` is committed to source control. `mvl build --locked` fails if the lock is stale (CI enforcement).

### 6. Assurance — Trust Score per Package in `mvl audit`

`mvl audit` reports:

```
Dependency    Version  License  Extern lines  MVL lines  Trust score
mvl_http      1.2.0    MIT      312           1 840      85.5%
mvl_tls       0.4.0    Apache   89            430        82.8%
bridge.rs     —        —        47            —          (local)

Total extern surface: 448 lines across 3 packages
```

**Trust score** = MVL verified lines / (MVL lines + extern lines). Higher is better.
`extern-rationale` is printed for any package where trust score < 100%, so users can read the justification in the dependency output.

`mvl audit --sbom` emits a CycloneDX SBOM JSON (see #56) combining `[dependencies]` and `[native]` entries with their versions, licenses, and extern counts.

## Package Directory Structure

At rest in `$XDG_DATA_HOME/mvl/pkg/` (global cache):

```
~/.local/share/mvl/pkg/
├── http/
│   └── 1.2.0/
│       ├── mvl.toml
│       ├── src/
│       │   ├── server.mvl
│       │   ├── request.mvl
│       │   ├── response.mvl
│       │   └── internal/
│       │       └── ffi.mvl
│       └── bridge.rs
└── tls/
    └── 0.4.0/
        └── ...
```

In project (local overrides win):

```
myproject/
├── mvl.toml
├── mvl.lock
└── .mvl/
    └── pkg/
        └── http/         ← override fetched from cache
```

## Rationale

- **No second manifest file** (`dependency.toml` would be redundant; `mvl.toml` already exists per ADR-0009).
- **Both visibility mechanisms** because they solve different problems: `pub`/private is fine-grained (one function), `internal/` is coarse-grained (one subsystem). Go's ecosystem shows `internal/` is understood at a glance.
- **Git-only first** because Phase 3 is still defining the language; a premature registry creates permanent obligations.
- **`mvl build` does everything** because separate install/build steps cause "works on CI but not locally" problems. Cargo proved the one-command model works.
- **Trust score** makes supply chain risk visible without blocking. Users can decide whether `85.5%` is acceptable; `mvl audit` doesn't refuse to build, it informs.

---

## Consequences

- `mvl.toml` gains `[dependencies]` and optional `[native]` tables.
- `extern-rationale` is required when any `extern "rust"` block exists in the package.
- The resolver gains a third resolution path: `pkg.*` → `$XDG_DATA_HOME/mvl/pkg/{name}/{version}/src/`.
- `internal/` enforcement is added to the resolver (reject cross-package access).
- `mvl build` gains fetch + lock steps before transpilation.
- `mvl audit` becomes the supply chain transparency command.
- `mvl.lock` is generated and should be committed.
