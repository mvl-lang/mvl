# ADR-0014: Mutation Testing Execution Model — Single Compile, Parallel Runs

**Status:** Accepted
**Date:** 2026-04-18
**Context:** Issue #210 introduces `mvl mutate`, native behavioral mutation testing. The core architectural question is how to execute N mutants against the test suite without paying N × compilation cost (Rust compilation is the bottleneck, not test execution).

---

## Decision

MVL mutation testing uses **single-compile, parallel-run execution**: all mutants are embedded in one compiled binary via env-var dispatch, then N test processes run in parallel each selecting a different mutant via `MVL_MUTANT=<id>`.

```
1 compilation  →  N parallel test processes  →  killed/survived per mutant
```

## The Three Execution Models (and why MVL chose the third)

| Model | Compilations | Parallelism | Notes |
|-------|-------------|-------------|-------|
| **Sequential** | O(N) serial | None | Simple; 50 mutants = 50 cargo builds (~minutes) |
| **Parallel workspaces** | O(N) parallel | Full | N temp Cargo dirs; high disk + memory pressure |
| **Single compile, env dispatch** | O(1) | Full (test processes only) | ✅ Optimal for Rust transpilation target |

## Mechanism

The transpiler, when invoked in mutation mode, emits a dispatch wrapper around every mutation point:

```rust
// Original: a + b
// Mutated output (mutation mode only):
{
    match ::std::env::var("MVL_MUTANT").as_deref() {
        Ok("m1") => a - b,   // ArithmeticFlip: + → -
        Ok("m2") => a * b,   // ArithmeticFlip: + → *
        Ok("m3") => a / b,   // ArithmeticFlip: + → /
        _        => a + b,   // Original (default)
    }
}
```

The runner then:
1. Compiles once: `cargo test --no-run`
2. Spawns N processes in parallel: `MVL_MUTANT=m1 ./test_binary`, `MVL_MUTANT=m2 ./test_binary`, ...
3. Killed = process exits non-zero (test failed); Survived = exits zero

## Behavioral-Only Mutations

MVL's type system pre-kills structural mutants (~70% of all possible mutations):

| Mutation category | Pre-killed by compiler? | Example |
|-------------------|------------------------|---------|
| Type changes (`Int` → `String`) | ✅ Type error | Compiler rejects |
| Null removal (`Option<T>` → `T`) | ✅ Type error | Compiler rejects |
| Effect removal (`!DB` → pure) | ✅ Effect checker | Compiler rejects |
| Taint change (`Tainted` → `Clean`) | ✅ IFC checker | Compiler rejects |
| Arithmetic flip (`+` → `-`) | ❌ Type-preserving | Tests must catch |
| Comparison flip (`<` → `<=`) | ❌ Bool-preserving | Tests must catch |
| Constant change (`0` → `1`) | ❌ Type-preserving | Tests must catch |
| Branch deletion (remove else) | ❌ Type-preserving | Tests must catch |
| Return value (`Some(x)` → `None`) | ❌ Type-preserving | Tests must catch |

The mutation generator only produces mutations in the second group. The compiler acts as a free pre-kill filter, making MVL mutation scores higher-signal than in untyped languages.

## Chunking for Large Mutant Sets

If the mutant count exceeds a threshold (default: 500), the binary grows large and env-var dispatch overhead increases. The runner batches mutations into groups of ≤500 per binary:

```
Mutants 1–500   → binary_batch_1 → 500 parallel runs
Mutants 501–1000 → binary_batch_2 → 500 parallel runs
...
```

Each batch pays one compilation. Total compilations = ⌈N/500⌉.

## Concurrency

Test processes run with a configurable worker pool (default: number of logical CPUs). The runner:
- Spawns workers using a semaphore-guarded channel
- Collects killed/survived results as processes complete
- Streams progress to stderr: `[42/150] ✓ m42 killed  △ m43 survived`

## Rationale

1. **Compilation is the bottleneck.** A single `cargo build` for a typical MVL program takes 2–5 seconds. With 50 mutants, sequential execution costs 100–250 seconds. Single-compile costs 2–5 seconds plus 50 parallel millisecond-range test runs.

2. **MVL controls the transpiler output.** The env-var dispatch wrapper is generated by the transpiler, completely transparent to the user. No mutation testing framework is needed at the MVL language level.

3. **Process isolation is sufficient.** Each test process is independent (no shared state between mutant runs). Env-var dispatch is the lightest possible isolation mechanism.

4. **Consistent with coverage model.** Coverage (#209, ADR-0010) uses a hidden `zzz_mvl_cov_report` test and atomic counters. Mutation testing follows the same principle: invisible to the user, transparent in the generated Rust.

## Consequences

- **Test suite must be deterministic.** Non-deterministic tests produce false survivals. The runner runs with `--test-threads=1` within each process (consistent with coverage).
- **Env-var collision.** MVL programs that read `MVL_MUTANT` themselves would interfere. This env var is reserved by the toolchain (documented alongside `MVL_COV_OUT`).
- **Binary size grows linearly.** Each mutation adds a small match arm. Chunking at 500 keeps binaries manageable.
